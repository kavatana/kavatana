# Startup Portfolio Map

This document gives a public, non-sensitive overview of the products I am
building. Source repositories are private because these are future startup
assets, but the product thinking, engineering discipline, and operating logic
are public enough for collaborators, mentors, recruiters, and investors to
understand the direction.

## Portfolio Principles

- Build for local Cambodian users first, then generalize only after learning.
- Treat every project as a product with users, risks, workflows, and operating
  constraints.
- Keep source code, data, credentials, and unfinished strategy private.
- Publish honest summaries, architecture notes, and case studies that do not
  leak implementation details.
- Use AI agents to speed up product engineering while keeping human review,
  tests, and claim discipline in the loop.

## Public vs Private Boundary

Public:

- Product positioning
- Architecture summaries
- Case studies
- Screenshots and demos when safe
- Learning notes and engineering process
- Verification notes: type checks, builds, audits, and manual smoke tests

Private:

- Source code for active startup products
- API keys, credentials, and deployment details
- Proprietary product strategy
- User data, partner data, and unreleased operating plans
- Security-sensitive implementation details

## Personal Systems as Proof

Studio OS is not one of the seven startup tracks. It is a personal command
center and life operating system that proves I can build a serious private tool,
ship a public demo safely, and explain the engineering tradeoffs behind it. The
public demo is intentionally privacy-safe; the source repository remains private.

- I can turn broad ideas into concrete product surfaces.

- I can use AI tools as part of a serious engineering workflow.
- I care about Cambodia-specific problems and Khmer-first access.
