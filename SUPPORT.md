# Support

Choose the route that matches the help you need. Never post credentials,
personal data, private startup source, unreleased strategy, or vulnerability
details in a public thread.

## Portfolio Bugs

Use the [portfolio bug form](https://github.com/kavatana/kavatana/issues/new?template=bug.yml)
for reproducible problems in the website, public documentation, or repository
automation. Include the affected public URL, expected behavior, reproduction
steps, and redacted evidence.

## Public Claim Corrections

Use the [claim correction form](https://github.com/kavatana/kavatana/issues/new?template=claim-correction.yml)
when a public statement is stale, unclear, or unsupported. Link to public-safe
evidence or state what the owner must verify; do not paste private evidence.

## Improvements

Use the [improvement form](https://github.com/kavatana/kavatana/issues/new?template=improvement.yml)
for a focused public-proof, accessibility, or maintainability outcome. Keep one
issue centered on one observable result.

## Security Reports

Use GitHub's [private vulnerability reporting](https://github.com/kavatana/kavatana/security/advisories/new)
and follow [`SECURITY.md`](SECURITY.md). Do not open a public issue for a
suspected vulnerability.

## Conduct Concerns

Follow [`CODE_OF_CONDUCT.md`](CODE_OF_CONDUCT.md) and use its private reporting
route for sensitive conduct concerns.

## Support Boundary

This repository is a personal portfolio and public proof surface, not a managed
service. Support is best-effort and no response-time or implementation guarantee
is promised. The maintainer cannot accept credentials, private repository
access, customer data, or responsibility for another project's security or
deployment decisions.
